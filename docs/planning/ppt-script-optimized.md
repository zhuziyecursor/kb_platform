### 第2页【通用大模型的三大局限】

- **核心观点**：大模型很强，但在企业垂直领域存在三大硬伤，直接用于业务问答风险极高
- **要点**：
  1. **知识更新不及时**：模型训练数据有截止日期，新制度、新法规、新项目经验无法覆盖
  2. **垂直领域理解浅**：审计、财务、合规等专业领域的术语、条款、上下文关联能力不足
  3. **企业知识无法触达**：内部制度、流程、项目经验不在模型训练语料中，回答只能靠"猜"
- **金句**："通用大模型知道'会计准则是什么'，但不知道'我们公司这条制度第3条是怎么写的'。"
- **视觉**：左侧大模型图标+三个红色警示标记，右侧简短文字，底部一条分隔线标注"企业知识 = 黑盒"

---

### 第3页【解决思路：本地模型 + 企业知识库（RAG）】

- **核心观点**：不是替代大模型，而是给大模型配一个"企业外脑"——先检索、再生成、带引用、可溯源
- **要点**：
  1. **RAG架构（检索增强生成）**：先检索企业知识，再让模型基于准确资料回答，不是凭空编造
  2. **本地可控**：模型选型、数据权限、回答策略企业自主掌控，敏感数据不出域
  3. **知识持续沉淀**：制度更新、项目复盘、经验总结持续入库，系统越用越准
  4. **答案可追溯**：每句话都有出处，用户可以点击跳转到原文档验证
- **视觉**：流程图 用户提问 → 查询理解 → 知识检索 → 上下文组装 → 模型生成 → 带引用回答

## 第二部分：现状与不足（3页）—— 重点展开

### 第4页【公司现有产品：小智助手的能力现状】

- **核心观点**：小智已经验证了"企业知识问答"的业务价值，但技术上属于**概念验证（PoC）级别**，存在明确的架构天花板
- **小智当前已具备的能力（客观陈述，不贬低）**：
  - 问答形态：单轮/多轮对话，SSE流式输出
  - 检索方式：**单路向量召回**（Dense Retrieval）
  - 功能特性：专项问答、智能检索、会话记录、复制/收藏/点赞
- **小智的技术实现方式**：
  - 文件解析：**直接调用大模型API解析文件内容**，无独立解析引擎
  - 数据入库：解析后的文本直接生成向量存入向量库
  - 知识检索：**纯向量相似度匹配**，无关键词检索、无重排序、无精排模型
  - 前端体验：简单的对话页面，**无文件解析过程可视化**
- **定性**：小智是需求验证器，证明了"审计行业需要知识问答"，但支撑不了"精准、可信、可治理"的企业级要求
- **视觉**：左侧小智功能清单（绿色勾），右侧"已验证价值"奖杯图标，底部一行小字"下一步：从PoC升级到企业级平台"

### 第5页【小智的核心瓶颈：四大技术债务】

- **核心观点**：小智的每一个技术选择，在业务规模扩大后都会变成"技术债务"

| 瓶颈维度 | 小智的实现方式 | 产生的具体问题 | 业务影响 |
|----------|---------------|---------------|---------|
| **文件解析** | 直接用大模型API解析文件，无独立解析引擎 | 大模型解析不稳定（长文本截断、格式丢失、表格乱码）；解析失败无感知；不支持扫描件/图片PDF | 上传的文档解析质量不可控，入库即带"硬伤" |
| **数据清洗** | **无清洗环节** | 页眉页脚、特殊符号、HTML标签、全角字符直接入库；无质量评分 | 向量库中充斥着噪声文本，检索时召回大量无关内容 |
| **数据分块** | **无分块策略** | 直接按固定长度或让大模型自行决定切分；无语义边界识别；无父子文档结构 | "切太大"→检索命中大段无关内容；"切太小"→丢失上下文，回答支离破碎 |
| **知识检索** | **单路向量匹配** | 仅依赖向量相似度，对"第几条""编号""专有名词"不敏感；无BM25关键词检索；无重排序模型；无FAQ缓存 | 精确条款查询经常答错；高频问题每次都要走完整链路（3-8秒）；成本高昂 |

- **视觉**：左侧四个瓶颈用红色高亮卡片纵向排列，右侧一个大箭头指向"KB-Platform升级路径"
- **金句**："小智回答了'有没有需求'，KB-Platform要回答'能不能放心用'。"

## 第三部分：KB-Platform 核心逻辑与能力（4页）

### 第7页【KB-Platform 项目总览：不是做一个功能，是搭建一个企业级知识平台】

- **核心观点**：KB-Platform 采用微服务 + 异步消息队列 + 契约优先的架构，是一条完整的"知识入库→精准检索→可信问答→可观测治理"的企业级链路
- **技术架构**：
  ```
  前端门户（Next.js 14）
    → kb-gateway（API网关，JWT鉴权、路由、trace_id注入）
    → Java微服务群：
        ingest-service（文档元数据、状态机、ACL、知识空间）
        vector-service（消费embed-task、Milvus upsert、状态回写）
        rag-service（检索编排：Query改写→Dense检索→ACL过滤→Rerank→Parent回捞→Prompt组装→LLM生成）
        llm-gateway（LLM统一代理：MiniMax阻塞/流式、审计日志）
        auth-adapter（OIDC登录、OBO Token Exchange）
        user-service（用户上下文、权限组）
    → Python服务群：
        kb-doc-processor（Tika解析 → TextCleaner清洗 → Chunker分块 → Embedding向量化）
        rerank-service（BGE-Reranker-v2-m3本地精排）
    → 基础设施：
        PostgreSQL（元数据、ACL、审计）+ Redis（缓存、会话）
        Milvus（向量库，1024维，HNSW索引）+ Kafka（消息总线）
        MinIO（对象存储，原始文件）+ Apache Tika（文档文本提取）
  ```
- **关键定性**：
  1. **微服务拆分**：每个服务职责单一，可独立扩展和升级
  2. **Kafka异步解耦**：文件上传后异步处理，削峰填谷，5分钟内可检索
  3. **契约优先**：OpenAPI + Kafka Schema + Milvus Collection 统一管控，接口变更可追溯
  4. **严格表所有权**：每个服务只写自己拥有的表，防止数据耦合
- **视觉**：分层架构图，不同颜色区分Java服务（蓝色）、Python服务（橙色）、基础设施（灰色）

### 第8页【知识库建设核心逻辑：文件上传 → 向量入库的完整流水线】

- **核心观点**：知识库不是"文件上传 + 向量入库"两步，而是 **10+步精密工程流水线**，每一步都影响最终检索质量

**完整链路**：
```
文件上传 → 权限校验 → MinIO存储 → Kafka(file-ingest) → kb-doc-processor
  → ① Tika多格式解析（PDF/Word/Excel/PPT/TXT/HTML/Markdown）
  → ② TextCleaner五层清洗（编码规范→去HTML→全角转半角→去页眉页脚→压缩空行）
  → ③ 质量评分（初始100分，HTML标签扣分，低质量文档标记告警）
  → ④ 智能分块（5种模式：HEAD_FIRST/TAIL_FIRST/UNIFORM/SMART/SMART_LLM）
  → ⑤ Parent-Child双层架构（Parent保存完整语义单元，Child作为检索单元）
  → ⑥ ChunkType语义推断（definition/procedure/rule/example/disclaimer）
  → ⑦ Embedding向量化（BGE-zh-v1.5，1024维，batch处理）
  → ⑧ 元数据展平（tags/sec_level/region_code/biz_domain/perm_group_id/effective_to）
  → ⑨ 持久化（knowledge_clean + knowledge_structured + embed_task）
  → ⑩ Kafka(embed-task) → vector-service → Milvus upsert → 文档READY
```

**与小智的关键差异**：

| 环节 | 小智 | KB-Platform |
|------|------|-------------|
| 解析引擎 | 大模型API（不稳定） | Apache Tika Server（工业级，支持50+格式） |
| 清洗环节 | **无** | 五层清洗 + 质量评分（0-100分） |
| 分块策略 | 无控制 | 5种模式 + Parent-Child + LLM精修边界 |
| 过程可视化 | **无前端页面** | 上传页展示完整流水线进度（解析→清洗→切片→向量化→入库） |
| 状态追踪 | 无 | DRAFT→PENDING→PROCESSING→READY/FAILED 状态机 |
| 失败重试 | 无 | 支持单文档重试，失败后保留错误原因 |

- **视觉**：横向流水线流程图，每个步骤用图标+文字，已完成的步骤用绿色，小智缺失的步骤用红色虚线框

### 第9页【智能问答核心逻辑：12步精密Pipeline】

- **核心观点**：一个高质量的企业问答，不是"向量检索 + 大模型生成"两步，而是 **12步精密协作**，每一步都有明确职责和容错机制

**完整RAG Pipeline**：
```
用户输入
  → ① 缓存检查（Redis，命中直接返回）
  → ② 会话上下文（读取历史对话，最多10轮，TTL 1800秒）
  → ③ 查询改写（同义词扩展 + 简单指代消解 + 条款关键词增强）
  → ④ 查询向量化（BGE-zh-v1.5，1024维）
  → ⑤ Dense检索（Milvus COSINE相似度，TopK=50，带tenant_id预过滤）
  → ⑥ ACL客户端过滤（sec_level/perm_group_id/effective_to，50条→可能剩15条）
  → ⑦ 知识空间过滤（按spaceId子树过滤docId）
  → ⑧ Rerank精排（BGE-Reranker-v2-m3，Top20→Top5）
  → ⑨ DB ACL二次校验（PostgreSQL doc_acl表精确校验）
  → ⑩ Parent回捞（根据parent_ref回捞Parent完整文本用于生成上下文）
  → ⑪ 拒答判断（无结果/低置信<0.3/无权限 → 委婉拒答）
  → ⑫ Prompt组装 + Token预算控制 + LLM生成 → 引用返回 → 会话保存 → Trace记录
```

**关键创新点**：
1. **三级拒答策略**：不是"不知道就瞎编"，而是明确区分"知识库没有"/"您没权限"/"召回质量差"
2. **Prompt Token预算控制**：上下文窗口8192 token，输入预算75%，历史15%，超长引用自动首尾压缩
3. **Parent-Child回捞**：检索用"子弹"（Child小块精确命中），生成用"炮弹"（Parent完整上下文）
4. **全链路Trace**：trace_id贯穿前端→网关→所有微服务→Kafka→Milvus→LLM，每一步耗时和状态可追溯

- **视觉**：Pipeline流程图纵向排列，不同颜色标注"已具备"（绿色）和"规划中"（橙色虚线）

### 第10页【行业四大痛点与我们的应对】

- **核心观点**：这些痛点不是我们这个项目独有的，是整个RAG行业的共识难题，但KB-Platform已有明确解法

| 痛点 | 现象 | 小智的表现 | KB-Platform的应对思路 |
|------|------|-----------|---------------------|
| **切片悖论** | 切大→噪声多、成本高；切小→语义碎 | 无分块控制，直接让大模型处理 | Parent-Child双层切片 + LLM精修边界 |
| **语义vs关键词** | 向量对专有名词/编号不敏感；关键词对同义词无解 | 纯向量检索，两种问题都存在 | 第二阶段多路召回（Dense+BM25+FAQ+RRF） |

| **模型幻觉** | 给对资料也可能胡说、捏造来源 | 无引用溯源，无法验证 | 三级拒答 + 引用校验 + 强制标注来源 |

## 第四部分：核心亮点（5页）

### 第11页【核心亮点一：双层渐进式智能分片 — 别人是"一刀切"，我们是"手术刀"】

- **核心观点**：文档切片不是技术细节，是检索质量的**第一决定因素**。小智"不切片"或"乱切片"，直接决定了回答质量的天花板。
- **KB-Platform的五层切片体系**：

| 层级 | 模式 | 作用 | 技术实现 | 适用场景 |
|------|------|------|---------|---------|
| L1 | HEAD_FIRST / TAIL_FIRST / UNIFORM | 基础固定长度切分 | 滑动窗口，512字符，10%重叠 | 简单文档、快速入库 |
| L2 | SMART（规则语义切片） | 按章节/条款边界切分 | 正则引擎识别标题（第X章/一、/1.1/总则/附则） | 制度文件、法规条文 |
| L3 | SMART_LLM（LLM精修边界） | LLM判定最佳切分点 | MiniMax输出段落分组索引，内容完全来自原文 | 复杂结构文档 |
| L4 | Parent-Child架构 | 小块检索 + 大块生成 | Parent（1500字符完整语义单元）+ Child（400字符检索单元） | 所有语义切片模式 |
| L5 | Fallback机制 | 保证可用性不降级 | LLM失败/超时/格式异常，自动回退到规则引擎 | 全场景兜底 |

- **与小智的对比**：
  - 小智："请大模型帮我把这个文件处理一下"→ 不可控、不可追溯、不可调优
  - KB-Platform："按语义边界精确切片，Parent保存完整上下文，Child用于精确检索"→ 可控、可追溯、可切换模式
- **业务价值**：同一条制度，问"第3条"能精确命中Child，问"这个制度讲了什么"也能召回Parent全文上下文
- **视觉**：左侧一页文档示意图标注"规则切分"和"LLM精修"，右侧对比表格，底部小智 vs KB-Platform对比卡片

### 第12页【核心亮点二：Parent-Child双层切片架构 — 用小钥匙开门，用大房间装答案】

- **核心观点**：解决"切片悖论"的核心设计——检索用"子弹"精确命中，生成用"炮弹"提供完整上下文
- **运作逻辑**：
  1. **入库阶段**：SemanticChunker识别章节标题 → 每个章节生成一个Parent（完整语义单元）→ Parent内部滑动窗口切出多个Child（检索单元）
  2. **检索阶段**：Child小块进入Milvus向量库，精确匹配用户query → 命中Child的parent_ref字段
  3. **生成阶段**：根据parent_ref回捞Parent完整文本 → 将Parent完整文本作为上下文喂给LLM
  4. **结果**：既保证检索精度（小块精确命中），又保证大模型有足够上下文生成高质量回答
- **技术细节**：
  - Parent最大1500字符，Child 400字符，Child重叠50字符
  - `parent_ref`格式：`docId/version/parentSeq`
  - Parent和Child都写入Milvus，Child用于检索，Parent用于生成
- **小智的缺失**：小智没有分块策略，更没有Parent-Child设计，检索要么召回超大段无关内容，要么召回碎片化文本导致回答断裂
- **视觉**：树状图展示Parent-Child关系，左侧小钥匙图标，右侧大房间图标

### 第13页【核心亮点三：全链路可观测性 — 别人回答错了不知道为啥，我们每一步都能追溯】

- **核心观点**：可观测性不是"日志多点"，而是从用户提问到答案返回的**每一个决策点都可回放、可量化**
- **可观测维度**：

| 可观测维度 | 具体内容 | 业务价值 |
|------------|---------|---------|
| **Pipeline Trace** | trace_id贯穿前端→网关→所有微服务→Kafka→Milvus→LLM | 回答错了？打开Trace抽屉，5分钟定位是召回问题还是生成问题 |
| **阶段耗时** | 改写/向量化/检索/ACL过滤/Rerank/Parent回捞/Prompt组装/LLM生成各阶段花了多少毫秒 | 首Token慢？看是Embedding慢还是LLM慢 |
| **召回详情** | 召回多少条、哪些文档、Rerank后保留几条、ACL过滤掉了多少 | 召回为空？看是Milvus没命中还是ACL全过滤了 |
| **Prompt预算** | 输入预算多少、实际用了多少、哪些引用被压缩/丢弃、历史对话保留了几轮 | 回答不完整？看是不是引用被压缩了 |
| **首Token时间** | 流式输出时，用户多久看到第一个字 | 体验指标，P95目标<1.5秒 |
| **拒答原因** | NO_MATCH / NO_PERMISSION / LOW_CONFIDENCE | 分析用户哪些问题是知识库覆盖不到的 |

- **小智的缺失**：小智无可观测体系，回答错了只能靠"用户反馈→人工猜测→逐个排查"
- **视觉**：模拟Trace抽屉的截图或手绘示意图，展示19个阶段的耗时瀑布图

### 第14页【核心亮点四：企业级权限模型 — 不是"能不能用"，而是"谁能看什么级别的内容"】

- **核心观点**：企业知识库的核心差异不是"检索准不准"，是"不能让不该看的人看到"
- **多维度ACL设计**：

| 维度 | 字段 | 示例 | 作用 |
|------|------|------|------|
| 租户隔离 | tenant_id | tenant_001 / tenant_002 | 数据物理隔离，Milvus检索带tenant过滤 |
| 密级控制 | sec_level | 1-5级（公开/内部/机密/绝密） | 低密级用户无法召回高密级文档 |
| 权限组 | perm_group_id | pg_audit_all / pg_finance_read | 按角色/部门/项目分组，避免Milvus表达式过长 |
| 地区 | region_code | BJ / SH / NATIONAL | 北京分公司员工只能看北京制度 |
| 业务域 | biz_domain | AUDIT / FINANCE / COMPLIANCE | 按业务线隔离 |
| 生效期 | effective_from / effective_to | 2024-01-01 ~ 2025-12-31 | 过期制度自动不可召回 |
| 细粒度ACL | doc_acl表 | USER/ROLE/DEPT三级授权 | 文档级精确控制 |

- **两道防线设计**：
  1. **Milvus预过滤**（快但metadata可能陈旧）：`tenant_id == X AND sec_level <= Y AND perm_group_id IN [...]`
  2. **PostgreSQL二次校验**（权威但慢）：`doc_acl`表精确校验userId/roleId/deptId
- **当前状态**：数据模型和字段已预留，ACL过滤逻辑已落地（客户端过滤），第三阶段接入真实JWT/OBO认证
- **小智的缺失**：小智仅有基础三级权限（公开/内部/机密），无权限组、无地区/业务域隔离、无生效期控制
- **视觉**：同心圆或漏斗图，越往里权限越严格，每一层标注一个维度

### 第17页【第二阶段（上）：为什么必须做多路召回？】

- **核心观点**：单路Dense检索有明确天花板，企业知识库**40-50%的查询是"精确编号+语义理解"的混合体**
- **6个真实Badcase**（基于第一阶段Pipeline Trace数据分析）：

| 问题场景 | 用户Query示例 | 第一阶段表现 | 根因 |
|----------|---------------|--------------|------|
| **精确条款查询** | "第十六条规定是什么" | 经常召回第十五条或第十七条 | Dense向量对数字不敏感，"第十六"和"第十五"向量相似度>0.95 |
| **专有名词/缩写** | "A-2024-09文件里的责任边界" | 召回不到或召回偏差 | 编号类token在embedding空间中缺乏区分度 |
| **高频重复问题** | "请假流程是什么" | 每次都走完整RAG链路（3-8秒） | 无FAQ缓存机制，成本高、体验差 |
| **一词多义** | "年假"vs"带薪休假"vs"年休假" | 同义词靠硬编码表，覆盖不全 | 规则同义词表无法覆盖业务术语演变 |
| **复杂追问** | "刚才说的那个，第三条具体怎么执行" | 指代消解失败，理解为新Query | 当前指代消解基于简单规则，覆盖不了多轮上下文 |
| **非知识问题** | "今天天气怎么样" | 强行走RAG，幻觉严重 | 无意图分类，所有问题都进知识库检索 |

- **架构师判断**：Dense检索解决"语义相近"问题，但企业查询是"精确匹配"+"语义理解"的混合体，单一路由天花板明显
- **视觉**：左侧6个Badcase卡片（红色背景），右侧一句话总结"必须升级为多路召回架构"

### 第18页【第二阶段（中）：六大功能详解】

- **核心观点**：每个功能都有明确的"问题→解法→收益"闭环

**功能1：BM25关键词召回**
- 解决：向量对编号、专有名词不敏感（如"第十六"和"第十五"向量太像）
- 方案：PostgreSQL自建倒排索引 + jieba分词 + 审计行业自定义词典
- 收益：精确条款Recall@5从45%→75%，编号类Query命中率>70%

**功能2：FAQ高置信短路**
- 解决：高频问题（如"请假流程"）每次都走完整RAG链路，3-8秒+消耗大量Token
- 方案：FAQ向量库 + cosine≥0.92直接返回答案，不走RAG
- 收益：高频问题延迟从3秒→100ms，LLM调用成本降为0
- FAQ来源：运营录入 + 自动抽取（从点赞高的会话中提取）

**功能3：条款编号Fast Path**
- 解决：用户问"第3条"，系统给一堆相关制度，就是不定位到第3条
- 方案：直接匹配section_path字段（如"3.2.1"、"第五条"），不走检索
- 收益：条款定位延迟<200ms，准确率>90%

**功能4：RRF多路融合**
- 解决：Dense、BM25、FAQ三路召回分数不可比（0.8 vs 12.5 vs 0.92），简单拼接会互相淹没
- 方案：Reciprocal Rank Fusion（k=60），只按排名融合，与分数绝对值无关
- 收益：整体Recall@10提升36%，检索稳定性提高（方差降低50%）

**功能5：LLM查询改写**
- 解决：同义词表覆盖不全、多轮指代失败、复杂问题不会拆分子Query
- 方案：MiniMax输出结构化JSON（mainQuery + subQueries + keywords + intent + contextSummary）
- 降级策略：LLM超时/格式错误/服务异常，自动回退到规则改写
- 收益：多轮对话准确率从40%→75%，复杂Query召回率+60%

**功能6：意图路由**
- 解决：闲聊/查通讯录/投诉都走RAG，成本高且幻觉严重
- 方案：规则Fast Path（延迟<10ms）+ LLM分类，分走6条链路：POLICY_QA / DOC_SEARCH / STRUCTURED_TOOL / CHITCHAT / HUMAN_HANDOFF / FALLBACK
- 收益：无效RAG调用减少60%，闲聊幻觉率<5%

- **视觉**：六大功能用六边形蜂窝图排列，每个配"问题图标→解法图标→收益数字"

### 第20页【第三阶段（上）：为什么必须做企业级治理？】

- **核心观点**：第一阶段能演示，第二阶段答得准，但没有第三阶段，正式推广会面临合规风险和安全事故

**直接上线的5个风险**：

| 风险 | 具体场景 | 后果 |
|------|----------|------|
| **越权访问** | 北京分公司员工看到上海分公司制度 | 数据泄露、合规事故 |
| **身份伪造** | 前端篡改header中的user_id | 冒充领导查询机密文件 |
| **无法追责** | 敏感制度被截图外传，查不到谁查的 | 安全事件无法溯源 |
| **成本黑洞** | 某用户高频调用，无人知晓 | 月度LLM账单暴涨 |
| **撤权滞后** | 员工离职后仍能查询历史制度 | 权限回收不及时 |

**架构师判断**：第一阶段的权限字段（tenant_id/sec_level/perm_group）是"数据模型正确"，但"认证链路不正确"。第三阶段不是重写权限，是把已有的权限数据模型接入真实的企业身份体系。

**前置条件**：不重建统一认证和通用网关，复用企业现有的BladeX OAuth2和基础网关，知识库侧只做协议适配和鉴权闭环。

### 第21页【第三阶段（中）：六大功能详解】

- **核心观点**：第三阶段不是"锦上添花"，是"基础设施"。没有评测，所有优化都是盲人摸象；没有审计，出了问题无法追责。

**功能1：统一认证 + OBO Token**
- 解决：BladeX token不能直接给资源服务用，权限范围太大；前端传header中的user_id可被篡改
- 方案：OIDC facade + RFC8693 Token Exchange。用户用BladeX登录→换user_access_token→再换obo_access_token（aud=mcp-kb，有效期5分钟，仅含知识库所需最小claims）
- 安全设计：Fail-Closed（上游不可用时默认拒绝）、Gateway+服务双重校验、JWKS密钥轮换、私钥不进镜像
- 收益：身份伪造风险降为0，token有效期从无限→5分钟

**功能2：用户上下文聚合（user-service）**
- 解决：用户权限信息分散在HR/OA/IAM多个系统，各服务各自解析逻辑不一致
- 方案：user-service统一聚合用户/部门/角色/密级/地区/业务域/权限组，Redis缓存5分钟，支持紧急撤权主动失效
- 收益：权限查询<50ms，撤权5分钟内全链路生效

**功能3：RAG权限过滤闭环**
- 解决：Milvus预过滤快但metadata可能陈旧，PG校验权威但全表扫描慢
- 方案：两道防线——Milvus标量预过滤（缩小候选集）+ PostgreSQL ACL二次校验（最终权威）。撤权时PG立即生效，Milvus metadata异步刷新
- 收益：越权召回率0%，撤权1分钟内生效

**功能4：评测体系（Eval Service）**
- 解决：改了Query改写逻辑，召回率提升了还是下降了？不知道。换了Embedding模型，回答质量变好了还是变差了？不知道
- 方案：评测数据模型（eval_dataset/case/run/result）+ 批量评测引擎 + LLM-as-Judge评分 + 人工复核抽检
- 评测指标：检索（Recall@K/MRR/nDCG）、生成（忠实度/完备度/引用准确率）、权限（越权率/误拒率）、性能（延迟P95/Token成本）
- 收益：版本发布前跑评测集，不达标不发布

**功能5：反馈闭环**
- 解决：前端已有点赞/点踩/收藏/报错按钮，但数据只是记录了，没有进入优化闭环
- 方案：点赞→提取QA对→运营审核→进入FAQ库；点踩/报错→自动生成Badcase→人工复核→进入评测集→驱动检索/生成优化
- 收益：反馈利用率>80%，每月自动发现5-10条高质量FAQ

**功能6：审计 + 成本治理**
- 解决：不知道谁查了机密文件、LLM花了多少钱、为什么上周成功率从99%跌到90%
- 方案：auth审计表 + resource审计表 + LLM调用日志，按tenant/user/space/model统计Token消耗，建立安全告警和性能告警
- 收益：安全事件分钟级发现，成本按租户实时统计，问题5分钟定位（trace回放）

**子里程碑拆解（6-8周）**：

| 子里程碑 | 周期 | 核心交付 |
|----------|------|----------|
| 3A：auth-adapter MVP | 1-2周 | OIDC discovery、JWKS、Token Exchange |
| 3B：Gateway与资源服务鉴权 | 1-2周 | 路由策略、iss/aud/scope/tenant校验 |
| 3C：用户上下文与权限组 | 1-2周 | user-service、/user/v1/context、perm_group计算 |
| 3D：RAG权限过滤与审计 | 1-2周 | Milvus ACL下推、PG二次校验、denied_ids审计 |
| 3E：评测、反馈与生产治理 | 2周 | Eval Service、反馈闭环、成本看板、监控告警 |

- **视觉**：六大功能按"安全→权限→评测→运营"分层排列，底部时间轴展示子里程碑

- **核心观点**：第三阶段不是写业务代码，是建基础设施。6-8周是保证质量的前提下最紧凑的周期。

**验收指标**：

| 安全 | 身份伪造抵抗 | 无 | 100%拒绝 | 渗透测试 |
| 权限 | 撤权生效时间 | N/A | <1分钟 | 撤权演练 |

| 成本 | Token消耗可视 | 无 | 实时仪表盘 | 看板验证 |
| 性能 | RAG成功率 | ~95% | >99% | 压测 |

**领导最可能问的三个问题**：

**Q1："第三阶段为什么需要6-8周？能不能压缩？"**
A：第三阶段不是写业务代码，是建基础设施。统一认证需要对接BladeX、管理密钥、做安全审计，每一环都不能出错。评测体系需要收集标注数据、校准评判标准，这些都需要时间。压缩可能在权限链路留下安全漏洞，得不偿失。

**Q2："如果资源有限，能否只做第二阶段？"**
A：第二阶段可以独立交付，产出是"更准、更快、更便宜"的问答体验。但如果要正式推广给业务部门，第三阶段必须跟上。否则权限没打通、审计没记录，出了问题无法追责，不符合企业合规要求。建议节奏：第二阶段完成后先内部试用，同时并行启动第三阶段。

**Q3："为什么不直接用Dify/阿里云百炼？"**
A：SaaS工具适合PoC，但审计行业需要密级控制、企业认证、审计合规、多租户隔离，这些在SaaS上无法深度定制。我们借鉴产品形态，核心链路自研可控。第三阶段的安全和评测能力，是SaaS工具无法提供的。

- **视觉**：上半部分验收指标对比表，下半部分三个Q&A卡片（对话框形式）

## 第六部分：未来展望（3页）

### 第23页【未来展望一：文件解析智能化（智能体 + MCP）】

- **核心观点**：把解析流水线的每个节点都变成可调度的智能体
- **当前**：Parser(Tika) → Cleaner → Chunker → Embedder → Indexer（硬编码流水线）
- **未来**：每个节点 = 智能体（Agent）
  - OCR Agent：扫描件、图表、跨页表格识别
  - Table Agent：表格结构化提取，保留行列关系
  - Chart Agent：图表内容理解，生成描述文本
  - Meta Agent：自动推断文档类型、业务域、密级、标签
  - 通过MCP（Model Context Protocol）统一调用
- **MCP价值**：标准化工具调用接口，不同智能体可按需插拔；大模型通过MCP调用OCR/表格/图表解析能力，像调用函数一样自然
- **与小智的对比**：小智"直接扔给大模型处理"，未来"每个环节都是可控、可替换、可评测的智能体"
- **视觉**：当前"硬编码流水线"vs未来"智能体编排"的对比图

| 层级 | 方法 | 指标 |

## 附录：汇报话术建议

### 开场白（第1-3页）
> "各位领导，今天汇报的是企业AI知识库平台KB-Platform的建设方案。当前小智助手已经验证了'审计行业需要知识问答'这个需求，但在实际使用中发现，小智的检索精度、权限管控、可观测性都存在明显天花板。KB-Platform不是否定小智，而是在小智的基础上，用企业级架构全面升级，从'能回答'走向'可信问答'。"

### 小智对比环节（第4-6页）
> "小智的技术方案是：文件直接给大模型解析→生成向量→向量匹配→生成答案。这个方案在PoC阶段没问题，但有四个硬伤：第一，大模型解析文件不稳定，长文本会截断，表格会乱码，而且解析过程完全不可见；第二，没有数据清洗，页眉页脚、特殊符号直接入库，检索时召回大量噪声；第三，没有分块策略，要么召回超大段无关内容，要么召回碎片化文本；第四，检索只有单路向量匹配，对'第几条''编号'完全不敏感。这些问题不是修修补补能解决的，需要重新设计架构。"

### 阶段规划环节（第16页）
> "我们设计了三阶段路线：第一阶段证明'能做'，已经基本完成；第二阶段解决'答得准'，4-6周；第三阶段解决'敢不敢用'，6-8周。为什么分三阶段？如果跳过第二阶段直接做权限，会出现'系统很安全，但回答总是错的'；如果跳过第三阶段直接推广，会出现'回答对了，但不该看的人也能看'。三阶段是工程上的自然分层，也是风险控制的必要节奏。"

### Q&A预备

**Q："第二阶段投入产出比最高的是哪个功能？"**
A："FAQ短路和条款Fast Path。这两个功能开发周期短（各约1周），但用户体验提升最直观。FAQ短路让高频问题从3秒降到100毫秒；条款Fast Path让用户问'第3条'时直接定位，不再'猜'。这两个功能不需要改动核心架构，是在现有链路上增加'快捷通道'，风险最低、收益最明显。"

**Q："为什么不直接在小智上改造，而要新建一个平台？"**
A："小智的架构是单体/Dify编排，所有功能耦合在一起。要加BM25需要改检索层，要加Parent-Child需要改入库层，要加权限需要改全链路——改动成本比新建还高。KB-Platform采用微服务架构，每个服务职责单一，后续升级、扩展、替换都更容易。"

**Q："项目完成后，小智会下线吗？"**
A："不会。小智作为轻量级问答入口可以继续运行，KB-Platform作为企业级知识库平台承接更复杂、更精准、更安全的场景。两者是互补关系，不是替代关系。"
