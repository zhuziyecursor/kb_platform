## 第2页【通用大模型的三大局限】

- **核心观点**：大模型很强，但在企业垂直领域存在三大硬伤
- **要点**：
  1. **知识更新不及时**：模型训练数据有截止日期，新制度、新法规无法覆盖
  2. **垂直领域理解浅**：审计、财务等专业术语、条款、上下文关联能力不足
  3. **企业知识无法触达**：内部制度、流程、项目经验不在模型训练语料中
- **金句**："通用大模型知道'会计准则是什么'，但不知道'我们公司这条制度第3条是怎么写的'。"
- **视觉**：左侧大模型图标+三个红色警示标记，右侧简短文字

## 第3页【解决思路：RAG + 企业知识库】

- **核心观点**：不是替代大模型，而是给大模型配一个"企业外脑"
- **要点**：
  1. **RAG架构**：先检索企业知识，再让模型基于准确资料回答，不是凭空编造
  2. **本地可控**：模型选型、数据权限、回答策略企业自主掌控
  3. **答案可追溯**：每句话都有出处，可点击跳转到原文验证
  4. **知识持续沉淀**：制度更新、项目复盘持续入库，越用越准
- **视觉**：流程图 用户提问 → 查询理解 → 知识检索 → 上下文组装 → 模型生成 → 带引用回答

## 第4页【公司现有产品：小智助手 — 验证了需求，但触达天花板】

- **核心观点**：小智已验证"企业知识问答"的业务价值，但属于**PoC级别**，技术上存在四大技术债务
- **小智当前能力（客观陈述）**：
  - 单轮/多轮对话，SSE流式输出
  - 专项问答、智能检索、会话记录、复制/收藏/点赞
- **小智的技术实现方式（问题根因）**：
  - **文件解析**：直接调用大模型API解析，无独立解析引擎，**无前端过程展示**
  - **数据入库**：解析后的文本直接生成向量存入向量库，**无清洗、无分块控制**
  - **知识检索**：**纯向量相似度匹配**，无关键词检索、无重排序、无精排模型
- **定性**：小智是需求验证器，证明了"审计行业需要知识问答"，但支撑不了"精准、可信、可治理"
- **视觉**：左侧小智功能清单，右侧四个红色"技术债务"卡片纵向排列

## 第5页【四大技术债务详解】

- **核心观点**：小智的每一个技术选择，在业务规模扩大后都会变成"硬伤"

| 瓶颈 | 小智的实现方式 | 产生的具体问题 | 业务影响 |
|------|---------------|---------------|---------|
| **文件解析** | 大模型API解析，无独立引擎 | 长文本截断、格式丢失、表格乱码；解析失败无感知；不支持扫描件 | 上传的文档解析质量不可控，入库即带"硬伤" |
| **数据清洗** | **无清洗环节** | 页眉页脚、特殊符号、HTML标签直接入库 | 向量库中充斥噪声，检索召回大量无关内容 |
| **数据分块** | **无分块策略** | 让大模型自行决定切分；无语义边界；无父子结构 | "切太大"→噪声多；"切太小"→上下文断裂 |
| **知识检索** | **单路向量匹配** | 对"第几条""编号"不敏感；无BM25；无重排序；无FAQ缓存 | 精确条款经常答错；高频问题每次3-8秒；成本高昂 |

- **底部过渡**："这些问题不是修修补补能解决的，需要重新设计架构。"
- **视觉**：表格形式，小智列灰色，问题列橙色，业务影响列红色

## 第8页【知识库建设核心逻辑】

- **核心观点**：知识库不是"文件上传+向量入库"，而是 **10步精密流水线**

```
文件上传 → MinIO存储 → Kafka(file-ingest)
  → ① Tika多格式解析（PDF/Word/Excel/PPT/TXT/HTML/Markdown，50+格式）
  → ② TextCleaner五层清洗（编码规范→去HTML→全角转半角→去页眉页脚→压缩空行）
  → ③ 质量评分（初始100分，低质量标记告警）
  → ④ 智能分块（5种模式：HEAD_FIRST/TAIL_FIRST/UNIFORM/SMART/SMART_LLM）
  → ⑤ Parent-Child双层架构（Parent=完整语义单元，Child=检索单元）
  → ⑥ ChunkType语义推断（definition/procedure/rule/example/disclaimer）
  → ⑦ Embedding向量化（BGE-zh-v1.5，1024维）
  → ⑧ 元数据展平（tags/sec_level/perm_group/region/biz_domain/effective_to）
  → ⑨ 持久化（knowledge_clean + knowledge_structured + embed_task）
  → ⑩ Kafka(embed-task) → vector-service → Milvus upsert → READY
```

- **与小智的关键差异**：
  - 小智："请大模型帮我把这个文件处理一下"→ 黑盒、不可控
  - KB-Platform：每一步都可观测、可配置、可重试，前端展示完整流水线进度
- **视觉**：横向流水线流程图，已完成的步骤绿色，小智缺失的步骤红色虚线框

## 第9页【智能问答核心逻辑】

- **核心观点**：一个高质量的企业问答，是 **12步精密Pipeline**，每一步都有明确职责

```
用户输入
  → ① 缓存检查（Redis命中直接返回）
  → ② 会话上下文（最多10轮，TTL 1800秒）
  → ③ 查询改写（同义词+指代消解+条款增强）
  → ④ 查询向量化（BGE-zh-v1.5，1024维）
  → ⑤ Dense检索（Milvus COSINE，TopK=50）
  → ⑥ ACL客户端过滤（sec_level/perm_group/effective_to）
  → ⑦ 知识空间过滤（按spaceId子树过滤）
  → ⑧ Rerank精排（BGE-Reranker-v2-m3，Top20→Top5）
  → ⑨ DB ACL二次校验（PostgreSQL doc_acl精确校验）
  → ⑩ Parent回捞（根据parent_ref回捞完整上下文）
  → ⑪ 拒答判断（无结果/低置信<0.3/无权限 → 委婉拒答）
  → ⑫ Prompt组装 + Token预算控制 + LLM生成 → 引用返回 → Trace记录
```

- **关键创新**：
  1. **三级拒答**：区分"知识库没有"/"您没权限"/"召回质量差"
  2. **Token预算控制**：上下文窗口8192，输入预算75%，超长引用自动首尾压缩
  3. **Parent-Child回捞**：检索用"子弹"精确命中，生成用"炮弹"提供完整上下文
  4. **全链路Trace**：trace_id贯穿前端→网关→所有微服务→Kafka→Milvus→LLM
- **视觉**：Pipeline流程图，已具备绿色，Phase 2新增橙色虚线

## 第10页【核心亮点一：双层渐进式智能分片 + Parent-Child架构】

- **核心观点**：别人是"一刀切"，我们是"手术刀式"精准切片；检索用"子弹"，生成用"炮弹"
- **五层切片体系**：

| 层级 | 模式 | 作用 | 适用场景 |
|------|------|------|---------|
| L1 | HEAD_FIRST/TAIL_FIRST/UNIFORM | 基础固定长度切分 | 简单文档、快速入库 |
| L2 | SMART（规则语义切片） | 按章节/条款边界切分 | 制度文件、法规条文 |
| L3 | SMART_LLM（LLM精修边界） | LLM判定最佳切分点 | 复杂结构文档 |
| L4 | **Parent-Child架构** | **小块检索 + 大块生成** | 所有语义切片模式 |
| L5 | Fallback机制 | LLM失败自动回退规则引擎 | 全场景兜底 |

- **Parent-Child运作逻辑**：
  1. 入库：SemanticChunker识别章节 → 生成Parent（1500字符完整语义）→ Parent内滑动窗口切Child（400字符检索单元）
  2. 检索：Child小块进入Milvus，精确匹配query
  3. 生成：根据`parent_ref`回捞Parent完整文本 → 喂给LLM生成高质量回答
- **与小智对比**：小智"不切片或乱切片"→ 检索要么召回超大段噪声，要么召回碎片化文本
- **视觉**：左侧文档示意图标注"规则切分"和"LLM精修"，右侧树状图展示Parent-Child关系
- **金句**："用小钥匙开门，用大房间装答案。"

## 第11页【核心亮点二：全链路可观测 + 企业级权限模型】

**左半：全链路可观测性**
- **核心观点**：别人回答错了不知道为啥，我们每一步都能追溯
- **可观测维度**：

| 维度 | 内容 | 业务价值 |
|------|------|---------|
| Pipeline Trace | trace_id贯穿全链路 | 回答错了？5分钟定位是召回问题还是生成问题 |
| 阶段耗时 | 改写/检索/Rerank/生成各阶段毫秒级 | 首Token慢？看是Embedding慢还是LLM慢 |
| 召回详情 | 召回多少条、Rerank后保留几条 | 召回为空？看是Milvus没命中还是ACL全过滤 |
| Prompt预算 | 预算多少、实际用了多少、哪些引用被压缩 | 回答不完整？看是不是引用被压缩了 |
| 首Token时间 | 流式输出用户多久看到第一个字 | 体验指标，P95目标<1.5秒 |

**右半：企业级权限模型**
- **核心观点**：不是"能不能用"，而是"谁能看什么级别的内容"
- **多维度ACL**：租户隔离(tenant_id) + 密级(sec_level) + 权限组(perm_group) + 地区(region_code) + 业务域(biz_domain) + 生效期(effective_to) + 文档级ACL(doc_acl)
- **两道防线**：Milvus预过滤（快，缩小候选集）+ PostgreSQL二次校验（权威，精确授权）
- **小智缺失**：仅有基础三级权限，无权限组、无地区/业务域隔离、无生效期控制
- **视觉**：左半模拟Trace抽屉瀑布图，右半同心圆权限漏斗图

## 第13页【第二阶段：多路召回 — 为什么必须做？怎么做？】

**上：为什么必须做？**
- **核心观点**：单路Dense检索有明确天花板，企业知识库**40-50%查询是"精确编号+语义理解"混合体**
- **6个真实Badcase**：

| 问题场景 | 用户Query | 单路Dense表现 | 根因 |
|----------|-----------|--------------|------|
| 精确条款 | "第十六条规定是什么" | 经常召回第十五或第十七条 | 向量对数字不敏感，相似度>0.95 |
| 专有名词 | "A-2024-09文件责任边界" | 召回不到或偏差 | 编号类token缺乏区分度 |
| 高频重复 | "请假流程是什么" | 每次走完整RAG（3-8秒） | 无FAQ缓存，成本高 |
| 一词多义 | "年假"vs"带薪休假" | 同义词靠硬编码，覆盖不全 | 规则表无法覆盖术语演变 |
| 复杂追问 | "刚才说的，第三条怎么执行" | 指代消解失败 | 规则覆盖不了多轮上下文 |
| 非知识问题 | "今天天气怎么样" | 强行走RAG，幻觉严重 | 无意图分类 |

**下：六大功能**

| 功能 | 解决什么问题 | 方案 | 收益 |
|------|-------------|------|------|
| BM25关键词召回 | 向量对编号、专有名词不敏感 | PG倒排索引+jieba分词+审计词典 | Recall@5: 45%→75% |
| FAQ高置信短路 | 高频问题每次都走完整RAG（3-8秒） | FAQ向量库，cosine≥0.92直接返回答案 | 延迟：3s→100ms，LLM成本降为0 |
| 条款编号Fast Path | 问"第3条"给一堆相关制度 | 直接匹配section_path，不走检索 | 延迟<200ms，准确率>90% |
| RRF多路融合 | 三路召回分数不可比 | Reciprocal Rank Fusion（k=60） | Recall@10提升36% |
| LLM查询改写 | 同义词覆盖不全、多轮指代失败 | 输出结构化JSON（mainQuery+subQueries+intent） | 多轮准确率40%→75% |
| 意图路由 | 闲聊/查通讯录都走RAG | 规则Fast Path+LLM分类，6条链路 | 无效RAG调用减少60% |

- **视觉**：上半部分6个Badcase卡片（红色），下半部分六大功能六边形蜂窝图

## 第14页【第二阶段：验收指标与投入产出】

- **核心观点**：第二阶段不是"做更多功能"，是"用数据证明召回质量真的提升了"

## 第15页【第三阶段：企业级治理 — 为什么必须做？怎么做？】

**上：为什么必须做？**
- **核心观点**：第一阶段能演示，第二阶段答得准，没有第三阶段，正式推广面临合规风险

| 风险 | 具体场景 | 后果 |
|------|----------|------|
| 越权访问 | 北京分公司员工看到上海分公司制度 | 数据泄露、合规事故 |
| 身份伪造 | 前端篡改header中的user_id | 冒充领导查询机密文件 |
| 无法追责 | 敏感制度被外传，查不到谁查的 | 安全事件无法溯源 |
| 成本黑洞 | 某用户高频调用，无人知晓 | LLM账单暴涨 |
| 撤权滞后 | 员工离职后仍能查询历史制度 | 权限回收不及时 |

**下：六大功能**

| 功能 | 解决什么问题 | 方案 | 收益 |
|------|-------------|------|------|
| 统一认证+OBO Token | BladeX token不能直接给资源服务用 | OIDC facade + Token Exchange，obo_token有效期5分钟 | 身份伪造风险降为0 |
| 用户上下文聚合 | 权限信息分散在HR/OA/IAM多个系统 | user-service统一聚合，Redis缓存5分钟 | 权限查询<50ms，撤权5分钟生效 |
| RAG权限过滤闭环 | Milvus预过滤快但可能陈旧 | 两道防线：Milvus预过滤 + PG二次校验 | 越权召回率0% |
| 评测体系 | 改了逻辑不知道召回率升了还是降了 | eval_dataset/case/run/result + LLM-as-Judge | 发布前跑评测集，不达标不发布 |
| 反馈闭环 | 点赞/点踩未进入优化闭环 | 点赞→FAQ库；点踩→Badcase→评测集→驱动优化 | 反馈利用率>80% |
| 审计+成本治理 | 不知道谁查了机密文件、花了多少钱 | auth/resource/LLM审计表 + 成本仪表盘 | 安全事件分钟级发现 |

- **子里程碑**：3A auth-adapter(1-2周) → 3B Gateway鉴权(1-2周) → 3C user-service(1-2周) → 3D RAG权限审计(1-2周) → 3E 评测反馈治理(2周)

**领导最可能问的三个问题**：

**Q1："第三阶段为什么需要6-8周？能不能压缩？"**
A：第三阶段不是写业务代码，是建基础设施。统一认证需要对接BladeX、管理密钥、做安全审计，每一环都不能出错。评测体系需要收集标注数据、校准评判标准，这些都需要时间。压缩可能在权限链路留下安全漏洞，得不偿失。

**Q2："如果资源有限，能否只做第二阶段？"**
A：第二阶段可以独立交付，产出是"更准、更快、更便宜"的问答体验。但如果要正式推广给业务部门，第三阶段必须跟上。否则权限没打通、审计没记录，出了问题无法追责，不符合企业合规要求。建议：第二阶段完成后先内部试用，同时并行启动第三阶段。

**Q3："为什么不直接用Dify/阿里云百炼？"**
A：SaaS工具适合PoC，但审计行业需要密级控制、企业认证、审计合规、多租户隔离，这些在SaaS上无法深度定制。我们借鉴产品形态，核心链路自研可控。第三阶段的安全和评测能力，是SaaS工具无法提供的。

## 附录：关键汇报话术

### 开场白（第1-3页，2分钟）
> "各位领导，今天汇报的是企业AI知识库平台KB-Platform的建设方案。当前小智助手已经验证了'审计行业需要知识问答'这个需求，但在实际使用中发现，小智的检索精度、权限管控、可观测性都存在明显天花板。KB-Platform不是否定小智，而是在小智的基础上，用企业级架构全面升级，从'能回答'走向'可信问答'。"

### 小智对比环节（第4-6页，4分钟）
> "小智的技术方案是：文件直接给大模型解析→生成向量→向量匹配→生成答案。这个方案在PoC阶段没问题，但有四个硬伤：第一，大模型解析文件不稳定，长文本会截断，表格会乱码，而且解析过程完全不可见；第二，没有数据清洗，页眉页脚、特殊符号直接入库；第三，没有分块策略，要么召回超大段无关内容，要么召回碎片化文本；第四，检索只有单路向量匹配，对'第几条''编号'完全不敏感。这些问题不是修修补补能解决的，需要重新设计架构。"

### 阶段规划环节（第12页，2分钟）
> "我们设计了三阶段路线：第一阶段证明'能做'，已经基本完成；第二阶段解决'答得准'，4-6周；第三阶段解决'敢不敢用'，6-8周。为什么分三阶段？如果跳过第二阶段直接做权限，会出现'系统很安全，但回答总是错的'；如果跳过第三阶段直接推广，会出现'回答对了，但不该看的人也能看'。三阶段是工程上的自然分层，也是风险控制的必要节奏。"

### Q&A预备

**Q："第二阶段投入产出比最高的是哪个功能？"**
A："FAQ短路和条款Fast Path。这两个功能开发周期短（各约1周），但用户体验提升最直观。FAQ短路让高频问题从3秒降到100毫秒；条款Fast Path让用户问'第3条'时直接定位，不再'猜'。"

**Q："为什么不直接在小智上改造，而要新建一个平台？"**
A："小智的架构是单体/Dify编排，所有功能耦合在一起。要加BM25需要改检索层，要加Parent-Child需要改入库层，要加权限需要改全链路——改动成本比新建还高。KB-Platform采用微服务架构，每个服务职责单一，后续升级、扩展、替换都更容易。"

**Q："项目完成后，小智会下线吗？"**
A："不会。小智作为轻量级问答入口可以继续运行，KB-Platform作为企业级知识库平台承接更复杂、更精准、更安全的场景。两者是互补关系，不是替代关系。"
