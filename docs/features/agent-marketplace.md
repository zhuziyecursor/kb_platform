# 技能广场 + 专家广场 产品与技术方案

## 背景与目标

将现有"扩展管理"页面（/extensions）拆分重构为两个独立模块：

- **技能广场**：工具/提示词能力的管理与发现，替换现有扩展管理
- **专家广场**：领域 AI 智能体的创建、发布与使用入口

两个模块均包含完整的发布审核流程（本期为仿真实现），为后期接入真实审批引擎预留结构。

---

## 一、产品设计

### 1.1 核心概念对比

| 模块 | 定位 | 类比 | 用户行为 |
|------|------|------|---------|
| **技能广场** | 工具/提示词的管理中心 | App Store 工具插件 | 浏览→开启→在对话中使用 |
| **专家广场** | 领域智能体的展示与使用 | App Store AI 应用 | 浏览→点击对话→直接使用 |

**关键区别**：技能是"能力原子"，专家是"能力组合体"（提示词 + 知识空间 + 技能的集合）。

**用户旅程**

```

```

**页面布局**

├─────────────────────────────────────────────────────────────────────┤
│  分类 Tab：全部 | 提示词类 | 工具类 | MCP类                          │
├─────────────────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│  │ 📋            │  │ 🔍            │  │ 📊            │             │
│  │ 审计报告生成  │  │ 法规检索助手  │  │ 异常交易识别  │             │

│  │ ● 已启用      │  │ ○ 已发布      │  │ ◑ 待审核      │             │

**技能状态流**

```
草稿 ──[提交审核]──▶ 待审核 ──[审核通过]──▶ 已发布 ──[开启]──▶ 已启用
  ▲                      │                            ←──[关闭]──┘
  └──────────[驳回]───────┘（附驳回原因）
```

状态徽章配色：草稿（灰）/ 待审核（橙）/ 已发布（蓝）/ 已启用（绿）

卡片按状态显示不同操作：

| 状态 | 操作按钮 |
|------|---------|
| 草稿 | 编辑 / 提交审核 / 删除 |
| 待审核 | 查看详情 / **审核通过** / **驳回** |
| 已发布 | 开启 / 详情 |
| 已启用 | 关闭 / 详情 |

**创建技能 Drawer（右侧滑出）**

```
Step 1：基本信息
  · 技能名称（必填）
  · 描述（必填）
  · 图标（emoji 选择器）
  · 分类（提示词类 / 工具类 / MCP类）

Step 2：技能配置（按分类显示不同表单）
  · 提示词类：提示词内容（大文本框）、支持变量 {变量名} 语法
  · 工具类：HTTP 接口 URL、请求方法、Headers、参数 Schema（JSON）
  · MCP 类：服务器类型（STDIO/HTTP）、命令/端点、环境变量

Step 3：预览（提示词类展示渲染效果）

底部：[保存草稿]  [提交审核]
```

**仿真审核说明**

技能处于"待审核"状态时，卡片上显示「审核通过」和「驳回」按钮（本期所有人可见，二期接入真实权限后仅管理员可见）：
- 点击「审核通过」→ 状态变为"已发布"，toast 提示
- 点击「驳回」→ 弹窗输入驳回原因，状态回到"草稿"

## 二、技术实现方案

### 2.1 数据存储策略

**本期（一期）：localStorage**

所有技能和专家的配置数据存 localStorage，利用现有 `useExtensions` hook 模式扩展。适用于单用户本地验证和产品形态演示。

**二期：迁移至后端服务**

新建 `agent-service` 持久化到 PostgreSQL（`kb_agent` schema），支持多用户共享、权限控制和真实审批流。届时前端只需将数据操作从 localStorage 切换为 API 调用，页面和组件逻辑无需重写。

### 2.2 数据结构

**专家（新增，存入 `kb_expert_agents`）**

```typescript
interface ExpertAgent {
  id: string;
  name: string;             // 合规审查专家
  description: string;      // 一句话描述
  icon: string;             // emoji
  category: string;         // 合规 / 审计 / 财务 / 税务 / 其他
  systemPrompt: string;     // 核心提示词
  spaceIds: string[];       // 绑定知识空间 ID 列表
  skillIds: string[];       // 绑定技能 ID 列表（预留）
  status: AgentStatus;
  publishNote?: string;     // 发布说明
  rejectReason?: string;    // 驳回原因
  createdAt: string;
  updatedAt: string;
}

type AgentStatus = 'DRAFT' | 'PENDING' | 'PUBLISHED' | 'LISTED';
```

**技能（在现有基础上扩展 status 字段）**

```typescript
// 现有 ExternalSkill / PromptConfig 新增：
status: SkillStatus;
rejectReason?: string;

type SkillStatus = 'DRAFT' | 'PENDING' | 'PUBLISHED' | 'ENABLED';
```

### 2.3 rag-service 最小改动（支持自定义 systemPrompt）

专家对话需要将专家的提示词透传给 rag-service，仅需改动两处：

**文件 1**：`ChatRequest.java` — 新增可选字段

```java
private String systemPrompt;  // 可选，非空时覆盖默认模板
```

**文件 2**：`PromptConstructionService.java` — 支持透传

```java
String sysPrompt = StringUtils.hasText(request.getSystemPrompt())
    ? request.getSystemPrompt()
    : buildDefaultSystemPrompt(docs, request.getLang());
```

改动量约 10 行，不影响现有 /rag 页面任何功能。

### 2.4 前端 API 层扩展

**文件**：`src/api/http-client.ts` 新增 `agentChat` 方法（对现有 `ragChat` 的语义化封装）

```typescript
agentChat(req: {
  tenantId: string;
  sessionId?: string;
  query: string;
  spaceId?: string;
  systemPrompt?: string;   // 新增：专家提示词
  lang?: 'zh' | 'en';
}): Promise<RagChatResponse>
```

## 六、二期规划（数据迁移至后端）

本期 localStorage 方案为产品形态验证，二期迁移时：

### 新增数据库表（kb_agent schema）

```sql
kb_agent.agent          -- 专家主表（含状态机字段）
kb_agent.agent_space    -- 专家绑定知识空间
kb_agent.agent_skill    -- 专家绑定技能
kb_agent.skill          -- 技能定义表
kb_agent.mcp_server     -- MCP Server 配置
kb_agent.agent_acl      -- 权限控制（按部门/角色/用户）
kb_agent.agent_version  -- 发布版本快照
```

### 新增服务

新建 `kb-mcp/agent-service`（端口 31006），提供 Agent / Skill / MCP Server 的 CRUD、发布审核、权限分配 API。

### 前端改动

`useAgents.ts` 和 `useExtensions.ts` 的数据操作从 localStorage 切换为 API 调用，页面和组件逻辑无需重写。

## 七、关键设计决策

| 决策点 | 本期选择 | 理由 |
|--------|---------|------|
| 数据存储 | localStorage | 无多用户需求，快速验证产品形态 |
| 审核流 | 仿真（前端状态机） | 体现产品完整性，后期接真实审批 |
| 对话接入 | 透传 systemPrompt 给 rag-service | 改动最小，复用全部现有 RAG 能力 |
| 多空间检索 | 暂取第一个绑定空间 | MVP 阶段，避免改动 MilvusSearchService |
| /rag 入口 | 保留，专家广场同时提供默认助手 | 不破坏现有用户习惯 |
| 技能与专家路由 | /skills 和 /experts 独立路由 | 职责清晰，二期各自可独立演进 |
