方案对比分析

方案A：RAGFlow 二次开发  vs  方案B：参考 RAGFlow 设计优化 kb-platform

评估日期：2026-05-19  |  基于 RAGFlow v0.25.4 源码分析  |  kb-platform main 分支

方案 A：Fork RAGFlow 二次开发

Fork RAGFlow 仓库，在其基础上添加 kb-platform 的差异化能力（精细 ACL、Pipeline Trace、反馈闭环）， 将向量存储从 ES/Infinity 替换为 Milvus，用 RAGFlow 的 React 前端替换 kb-portal。
需维护 Go 31K+ 行 大规模重构 ES→Milvus 替换 ACL 反向移植 放弃已有投资

方案 B：参考 RAGFlow 设计，渐进优化 kb-platform

用 MinerU API 替换文档解析层，参考 RAGFlow 的元数据提取/知识图谱/自动标签来增强现有管道。 保留 kb-platform 的 ACL、混合检索、Trace、反馈闭环体系，分 4 步渐进交付。
低风险，渐进式 无需新语言 保留 ACL 体系 约 200 行就能接入 MinerU 知识图谱需自建

五、优劣势拆解

方案 A 优势

  • 文档解析能力最强：MinerU + DeepDoc + Docling + PaddleOCR 四引擎
  • 知识图谱/GraphRAG 开箱即用
  • RAPTOR 多粒度树形检索内置
  • 社区活跃，持续有新特性
  • 有现成的 React 前端

方案 A 劣势

  • 必须维护 Go 后端 ~31K 行，需新技能
  • ACL 需从零反向移植，RAGFlow 仅有基础 tenant 隔离
  • 管道追踪 & 反馈闭环需重写
  • Milvus 不在支持列表，需完整实现 DocStoreConnection ABC
  • 全量一次性迁移，风险高，回滚困难
  • Python 代码有技术债（重复函数、超长方法、混合关注点）
  • 每次上游更新需重新 merge + 验证 fork 差异
  • 基础设施不兼容（ES vs Milvus, MySQL vs PG）

方案 B 优势

  • kb-doc-processor 仅 ~2900 行，Kafka 契约边界清晰，精确替换
  • 保留 ACL 体系，零改动
  • 保留混合检索（Sprint 3 已实现），零改动
  • 保留 Pipeline Trace + 反馈闭环，零改动
  • 保留 llm-gateway，零改动
  • 渐进式交付，每步独立上线，随时可停
  • 无需学习新语言
  • 可复用 RAGFlow Python 模块（NER/图/标签逻辑可直接引用）

方案 B 劣势

  • 知识图谱需从零构建（最大单项工作量，~3-4 周）
  • RAPTOR 树形检索不内置，需独立实现
  • 没有现成前端组件库
  • BM25 仍是 PG tsvector 原型，需自行升级
  • MinerU 仅解决 PDF，其他格式仍需 Tika（合理的分工）
六、最终结论

推荐方案 B：参考 RAGFlow 设计，渐进优化 kb-platform

kb-platform 的核心瓶颈是"文档解析质量"，不是"检索架构"
用一个 200K+ 行的全栈项目替换 2900 行的解析服务，投入产出比极差

核心决策逻辑：

  1. 唯一真正的短板是文档解析 — Apache Tika 对复杂 PDF（表格、多栏、公式）的解析质量远不如 MinerU。而 kb-doc-processor 通过 Kafka 契约与下游完全解耦，替换它只需 200-400 行新代码。
  2. ACL、混合检索、Trace、反馈闭环是真正的差异化资产 — RAGFlow 在这些方面不如 kb-platform。方案 A 需要把这些全部反向移植，工作量大且没有价值创造。
  3. 知识图谱是唯一的"新构建"项 — 方案 A 开箱即用，方案 B 需自建。但 RAGFlow 的 graphrag/ 是纯 Python，可直接参考复用其 NER + 社区检测逻辑。这不是从零开始。
  4. Go 31K 行是不可忽视的维护负债 — RAGFlow 的 Go API 层不可剥离，团队必须长期维护 Go 代码并跟上上游更新。
